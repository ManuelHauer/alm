## 2. GIF preservation test

The handoff §5.3 / §M1.6 requires uploading an animated GIF and verifying
that the generated resize variants preserve animation. This was the main
unknown going into M1.

### Method

1. The seed script generates a **real 3-frame animated GIF** at runtime
   using Sharp's `join.animated` API. Source dimensions: **1600 × 1200**
   — chosen so all `imageSizes` (400, 1200, 2400) actually trigger:
   - thumbnail 400 → downscale (real test)
   - medium 1200 → downscale (real test)
   - large 2400 → upscale, skipped by Sharp's default `withoutEnlargement`

   Frames: orange `#E8531E` → green `#1EC850` → blue `#1E50E8`
   Delay: 300 ms / frame
   Loop: 0 (infinite)

2. Seed uploads the GIF via the Payload Local API. Payload's
   `imageSizes` pipeline runs Sharp on the original and writes resize
   variants to `/media/`.

3. Inspected each generated file with
   `sharp(file, { animated: true }).metadata()` to count frames.

4. Cross-checked by serving each variant over HTTP through the running
   dev server (`/api/media/file/<filename>`) and re-running the metadata
   inspection on the response body — confirms not just the file on disk
   but also the bytes Payload actually serves.

### Result

| File served | pages | loop | delay | size |
|---|---|---|---|---|
| `seed-animated.gif` (original) | **3** | 0 | [300, 300, 300] | 1600×3600 |
| `seed-animated-1200x900.gif` (medium) | **3** | 0 | [300, 300, 300] | 1200×2700 |
| `seed-animated-400x300.gif` (thumbnail) | **3** | 0 | [300, 300, 300] | 400×900 |

**Sharp 0.34.2 preserves animation through Payload's resize pipeline
without any workaround.** All three variants — including the heavily
downscaled 400 px thumbnail — serve real 3-frame animated GIFs over HTTP.

> The `1600×3600` reported height for the original is Sharp's
> animated-mode convention: it stacks the 3 frames vertically (3 × 1200).
> The actual frame size is correct.

The handoff §5.3 concern (resize variants flattened to first frame)
**does not apply** with the current Sharp version. The frontend can use
any size variant for GIFs without losing animation.

We still tag uploads with `isAnimatedGif: true` (via `handleGifUpload`)
so the frontend can make GIF-aware decisions (e.g. skip lazy-load
placeholders that would freeze the first frame). The flag also acts as
a tripwire: if a future Sharp upgrade regresses, we have a quick path
back to "always serve the original".

### Verification commands

```bash
# Inspect files on disk
node -e "
const sharp = require('sharp');
const files = [
  'media/seed-animated.gif',
  'media/seed-animated-1200x900.gif',
  'media/seed-animated-400x300.gif',
];
(async () => {
  for (const f of files) {
    const m = await sharp(f, { animated: true }).metadata();
    console.log(f, '→ pages:', m.pages, 'loop:', m.loop, 'delay:', m.delay);
  }
})();
"

# Inspect bytes served over HTTP
for u in /api/media/file/seed-animated.gif \
         /api/media/file/seed-animated-1200x900.gif \
         /api/media/file/seed-animated-400x300.gif; do
  out=$(mktemp).gif
  curl -sS "http://localhost:3000$u" -o "$out"
  node -e "require('sharp')('$out',{animated:true}).metadata().then(m => console.log('$u →', 'pages:', m.pages))"
  rm "$out"
done
```

## 4. How to run the project locally

### One-time setup

```bash
# Install toolchain
brew install node@22 pnpm postgresql@16 gh
brew link --force --overwrite node@22
brew services start postgresql@16

# Create database + role
psql postgres -c "CREATE ROLE alm WITH LOGIN PASSWORD 'alm';"
psql postgres -c "CREATE DATABASE alm OWNER alm;"

# Clone + install
git clone https://github.com/ManuelHauer/alm.git ~/Code/alm
cd ~/Code/alm
pnpm install

# Create .env (NOT committed)
cat > .env <<EOF
DATABASE_URI=postgres://alm:alm@localhost:5432/alm
PAYLOAD_SECRET=$(openssl rand -hex 32)
EOF
```

### Daily dev loop

```bash
cd ~/Code/alm
pnpm dev                              # Next.js + Payload at http://localhost:3000
# admin:           http://localhost:3000/admin
# REST API:        http://localhost:3000/api/entries
# M2 prototype:    http://localhost:3000/m2-prototype
```

### Seed the database (idempotent — wipes content collections, re-creates)

```bash
pnpm seed
# Creates: 1 admin, 2 folios, 5 media (incl. real animated GIF), 5 entries, 5 studio pages
# Admin login: admin@alm.local / almadmin1
```

### Other useful commands

```bash
pnpm generate:types                   # regenerate src/payload-types.ts after schema change
pnpm generate:importmap               # regenerate admin import map
pnpm lint                             # eslint
pnpm test:int                         # vitest integration tests
```

See `docs/DEPLOYMENT_NOTES.md` for the full local + production environment
reference (Postgres setup, Docker plan for M6, backup commands).
